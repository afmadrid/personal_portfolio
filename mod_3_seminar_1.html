Scrum Agile and Security Process


Security Engineering Processes

According to Sharma & Bawa, 2020, the most popular and accepted Security Processes (SE) processes around the world are:

SE Process Description
Microsoft’s Secure Development Lifecycle (SDL) This software development process has basically been derived from the classical spiral model and has been used for reducing software security related bugs.
Comprehensive Lightweight Application Security Process (CLASP) It is a framework that allows the user of the computer system to specify their security assurance and functional requirements.
Cigital Touchpoints (CT) Lightweight best practices that are applied to various software artifacts. It improves the various security and quality aspects of the end product.
Common Criteria (CC) Security Engineering process that originates from the OWASP.

Table 1. Most popular Security Engineering processes.


Security Practices Through Agile SCRUM Process

According to the study performed by Sharma & Bawa, 2020, next security practices are recommended at each phase of the Agile SCRUM process:

Phase Security Practices Information SE Process
Pre-requirements Initial Education Initial education is essential for the success of the project, it makes the team aware about the importance of security. CLASP, Miscrosoft SDL
Requirements Role Matrix All user roles and their access level are identified, it plays an important role for authentication and authorization. CLASP, Miscrosoft SDL, Cigital Touchpoints, Common Criteria
Security Requirements Identify security and privacy functionality, provides an insight about the project security concerns.
Agree on Definitions Defines stakeholders and agrees on a common set of security definitionsalong with organizational security policies and vision.
Identify resources and trust boundaries Defines system architecture form a network view, identifies data resources that a program can use and define trusted and untrustworthy entities interaction.
Specify Operational Environment Documents the operating environment’s assumptions and requirements to assess the impact on security.
Design Secure Design Principles By implementing security design principles and security risks in third party components are identified. CLASP, Cigital Touchpoints, Common Criteria
Assumption Documentation Recognize potential attacks and clearly document assumptions in order to assess the effect on security.
Requirements Inspection Its objective is to evaluate the quality of the requirements and outcomes of the team.
Risk Analyses Discover architectural flaws and prioritize them so that suitable mitigation is implemented.
Implementation Coding Rules Unsafe functions are determined and replaced with safer options. Miscrosoft SDL, Cigital Touchpoints
Security Tools Describes and publishes a list of authorized security tools to support the project.
Static Code Analyses Using tools for static analysis to scan the source code and find prevalent vulnerabilities.
Testing Dynamic Analysis Used track memory corruption, user privilege issues and other critical safety issues. CLASP, Miscrosoft SDL, Cigital Touchpoints
Security Testing Offers a useful insight into the software working in its real environment by simulating real working conditions and attack patterns.
Vulnerability and Penetration Testing
Release Operational Planning and Readiness Involves process documentation such as user manuals writing and security architecture documentation. CLASP, Miscrosoft SDL, Common Criteria
Signing the Code Code is signed and security regulations are approved.
Incident Response Planning Offers a response checklist that provides clear action rules for a security emergency.
Repository Improvement Quality of elements already in the repository can be improved.

Table 2. Secuirty processes through Agile Scrum SDLC.
